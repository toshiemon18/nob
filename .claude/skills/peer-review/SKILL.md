# peer-review

実装者と異なるコンテキストを持つ別エージェントにレビューさせて、見落としや偏りを減らす。
**plan（設計）とコード（実装）の両方**を対象にできる。

## 起動

```
/peer-review [args]
```

引数（自然言語で渡されるので柔軟に解釈する）:
- 対象の種別（例: 「plan をレビュー」「コード差分をレビュー」） — 省略時はコード差分
- 比較対象（例: 「main との差分」「直近の変更」「特定ファイル」「`docs/plans/xxx.md`」）
- 注力観点（例: 「テスト網羅性」「設計の一貫性」など）

不明な場合は user に最大 1 問だけ確認する。

## モード判定

呼び出し元または引数から **plan レビュー** か **コードレビュー** かを判定する。

- `plan=<path>` が渡された、または「plan をレビュー」と指示された → **plan レビューモード**
- それ以外 → **コードレビューモード**（従来の動作）

---

## plan レビューモード

### 手順

1. **対象の確定**
   - 指定された plan ファイルを Read する
   - `docs/design.md` があれば合わせて Read し、整合観点として使う

2. **レビューの実行**
   - `Agent(subagent_type=general-purpose)` を 1 回呼ぶ
   - prompt は自己完結。最低限以下を含める:
     - レビュー対象は plan ドキュメントであることを明記
     - plan ファイルのパスと全文
     - レビュー観点（plan 専用）:
       - 目的・背景の明確さ（なぜやるか、何を解決するか）
       - 設計の一貫性（既存コードや design.md との整合）
       - 設計の網羅性（エッジケース、失敗時の振る舞い、依存関係）
       - TODO の粒度（TDD で進められる粒度か、抜け漏れはないか）
       - スコープの妥当性（大きすぎ／小さすぎないか）
     - 出力フォーマット（コードレビューと同じ Critical / Important / Nice-to-have / Out of scope）
     - 「実装は禁止、評価のみ」を明記
     - 上限文字数の目安（600 字）

3. **結果の整形と提示**
   - agent の出力をそのまま user に提示
   - Critical 件数を 1 行で先頭に要約

4. **保存先（呼び出し元から指示があれば）**
   - plan ファイルに「レビューフィードバック」セクションがあれば、そこに追記する
   - 単体起動時は保存しない

## コードレビューモード

### 手順

1. **対象の確定**
   - `git status` / `git diff` / `git log` で変更内容を把握
   - 比較対象が指定されている場合はそれに従う
   - コンテキスト文書（plan, 設計など）が指定されていれば Read してから渡す

2. **レビューの実行**
   - `Agent(subagent_type=general-purpose)` を 1 回呼ぶ
   - prompt は自己完結。最低限以下を含める:
     - レビュー対象の概要（1〜2 文）
     - 変更ファイルのパス一覧
     - 必要ならコンテキスト文書のパスを示し、Read するよう指示
     - レビュー観点（標準セット）:
       - 設計／既存コードとの整合
       - テストの網羅性（境界、異常系、副作用）
       - 命名・構造・重複
       - エラー処理・セキュリティ
       - パフォーマンス上のリスク
     - 出力フォーマット:
       ```
       ### Critical
       - <指摘> (file:line) — <理由 1 行>
       ### Important
       - ...
       ### Nice-to-have
       - ...
       ### Out of scope
       - ...
       ```
     - 「実装は禁止、評価のみ」を明記
     - 上限文字数の目安（600 字）

3. **結果の整形と提示**
   - agent の出力をそのまま user に提示
   - Critical 件数を 1 行で先頭に要約

4. **保存先（呼び出し元から指示があれば）**
   - 呼び出し元（例: dev-workflow）から「結果を `<path>` に追記して」と渡されたら、対象セクションに append する
   - 単体起動時は保存しない（user が必要なら自分で保存する）

---

## 守ること

- 採用するレビューエージェントは **書き換え禁止**（実装してはいけない）
- 出力にコード提案を含めるかは観点次第。基本は「指摘 + 理由」のみで、提案は呼び出し元 / Refine フェーズの責務とする
- 指摘ゼロでも構造化された出力を返す（「Critical: なし」と明記）

## 単体利用の例

- `/peer-review` — 作業ツリー＋直近コミットを標準観点でレビュー
- `/peer-review plan=docs/plans/2026-04-30_core-structure.md` — plan をレビュー
- `/peer-review main との差分でテスト網羅性を中心に` — 観点を絞ってコードレビュー
