# Phase: IMPLEMENT

`docs/plans/<slug>.md` の TODO に従って実装する。t-wada 流の TDD（テスト駆動開発）を厳格に適用する。

## ゴール

- TODO がすべて green、テストが通っている
- リファクタリング後のコードがクリーンである
- plan の TODO 状態（チェックボックス）が現実と一致している

## TDD の原則（t-wada 流）

- **TODO リスト駆動**: 着手前に必ず plan の TODO を確認。新しく気づいたものはその場で TODO に追記する
- **Red → Green → Refactor の小さなサイクル**:
  - **Red**: 失敗するテストを 1 つ書く（コンパイルエラーも Red）
  - **Green**: 失敗テストを通す最小の実装。仮実装（fake it）で良い場合は躊躇わずに使う
  - **Refactor**: 重複・命名・構造のみを変える（振る舞いは変えない）。テストが緑のうちに行う
- **段階的一般化**:
  1. **仮実装** (`return 42` など定数を返す)
  2. **三角測量** (2 つ目のテストケースを足して一般化を強制する)
  3. **明白な実装** (もう仮実装で逃げる必要がない場面では直接書く)
- **歩幅は不安と相談**: 不安が大きいタスクほど小さく刻む。慣れた領域は大胆に進める
- **コミット境界**: Green になった瞬間と Refactor 後をそれぞれ commit 候補と捉える（user が明示的に commit を求めたときだけ実際にコミットする）

## 手順

1. **状態確認**
   - `docs/plans/<slug>.md` を Read。`status` が `implementing` であることを確認（違えば user に通知し、`status` の進め方を確認）

2. **TODO の先頭を選ぶ**
   - 一番上の未チェック項目を取る
   - 着手前に「何を Red で書くか」を 1 文で言語化

3. **Red**
   - 失敗するテストを書く（最小単位）
   - テストを実行して失敗を確認（成功してしまったらテスト自体が誤っている可能性 → 見直す）

4. **Green**
   - 仮実装でよければ最短の実装で通す
   - 「明白な実装」が即座に書ける場合はそれで構わないが、不安があれば仮実装に戻る

5. **テスト全体を実行**
   - 既存テストの破壊がないことを確認

6. **Refactor**
   - 重複の除去、命名の改善、抽象の引き上げ
   - 振る舞いは変えない。各リファクタの後にテストを再実行する
   - リファクタ余地が無ければスキップ

7. **TODO を更新**
   - plan の該当チェックを完了 (`- [x]`) に
   - 新しく気づいた task を末尾に追加（「途中で見つけた仕事」を漏らさない）

8. **次の task へ**: 2 に戻る

9. **全 task 完了**
   - plan の `status` を `reviewing` に更新、`updated` を当日に
   - user に「Implement が完了しました。Review に進みますか？」を 1 行で伝える

## 守るべきガードレール

- **テストなしの実装変更を行わない**（spike 用の探索コードは別。最終成果には残さない）
- **テストを skip / pending したまま完了扱いにしない**
- **設計が大きく崩れたら停止して user に相談**。plan を直すべきか、実装で吸収するかの判断を仰ぐ
- **TODO に無いコミットメントを入れない**（無駄なリファクタ、関係ない改善）。気づいたものは TODO に追加して別タスク扱い

## ツールの使い方

- ファイル変更は Edit / Write、検索は Grep / Glob を優先
- Bash は実行系（テスト、ビルド、ファイル操作）に限る
- 大規模な探索や別観点の分析は `Agent(subagent_type=Explore)` に短く委譲してよい
