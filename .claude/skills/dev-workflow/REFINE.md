# Phase: REFINE

Review で得られたフィードバックを、優先順位に従ってコードに反映する。リファクタリングを兼ねる。

## ゴール

- Critical と合意された Important の指摘がコードに反映されている
- 反映しなかった指摘について、理由が plan に記録されている
- テストが引き続き green である

## 優先順位の基準

- **Critical**: バグ、セキュリティ問題、設計の重大な逸脱 → 必ず対応
- **Important**: 重複・テスト不足・命名の混乱・将来の保守性に響くもの → 原則対応。スコープから外す場合は明示的に判断
- **Nice-to-have**: 好み・微改善 → user に確認してから対応するかを決める
- **Out of scope**: plan 外の発見 → 対応せず、後続の plan に切り出す候補として記録

## 手順

1. **状態確認**
   - `docs/plans/<slug>.md` を Read。`status` が `refining` であること
   - 「レビューフィードバック」セクションを構造化（Critical / Important / Nice-to-have / Out of scope）

2. **対応方針を作る**
   - 各指摘について「対応する／しない／後続 plan に切り出す」を判断
   - 判断結果を一時的に箇条書きで user に提示し、合意を取る（特に「対応しない」項目は必ず確認）

3. **対応する**
   - **TDD を維持**: 指摘がテストで保護できるものなら、まず Red → Green → Refactor のサイクルで進める
   - 純粋なリファクタ（振る舞いを変えない命名・抽出・重複除去）はテストを増やさず行ってよいが、実施前後でテストを必ず実行する
   - 1 つの指摘を 1 commit 候補として小さく進める

4. **plan ファイルを更新**
   - 各指摘の末尾に対応結果を追記:
     - `→ 対応済 (commit/ファイル名)`
     - `→ 見送り（理由）`
     - `→ 後続 plan へ: <新 slug 案>`
   - `updated` を当日に

5. **テスト全実行**
   - 全テスト green を確認

6. **次フェーズ移行**
   - `status` を `recap_pending` ではなく直接 `recap` 用にしたい場合、ここで `status: refining` のままにせず、Recap 起動条件として「Refine の対応一覧が plan に書き終わっていること」を満たす
   - user に「Refine が完了しました。Recap に進みますか？」を伝える
   - Recap を開始する直前に `status` を `recap` ではなく、plan の最終フェーズ用に `status: refining` のままで良い（Recap が `done` に進める）

## 注意

- 「ついでの改善」を入れない。スコープ外の指摘は plan の Out of scope に残す
- 複数の指摘を 1 度のリファクタで一気に解決したい場合でも、テストを跨ぐ変更は避け、ステップごとに green を保つ
