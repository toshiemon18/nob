# Phase: RECAP

最初に作成した plan と、最終的な実装の差分を見直す。乖離があれば、その経緯を plan に残しておくことで、後から「なぜそうしたか」が分かるようにする。

## ゴール

- plan の「設計」「TODO」と実装の差分が言語化されている
- 乖離した箇所には経緯（なぜその判断をしたか）が記録されている
- plan の `status` が `done` になっている

## 手順

1. **状態確認**
   - `docs/plans/<slug>.md` を Read

2. **差分の洗い出し**
   - plan の「設計」セクションに記載した API / モジュール構成と、実コードを比較
   - plan の「TODO」と実際のコミット粒度を比較
   - 必要なら `Agent(subagent_type=Explore)` に「plan のこの設計と実装の差分を 200 字以内で要約」のように短く依頼してもよい

3. **差分の分類**
   - **意図的な変更**: Refine や Implement 中の発見で plan を超えた判断 → 経緯を記録
   - **想定外の追加**: plan に無かったが実装で必要になったもの → なぜ必要になったか記録
   - **削除・スキップ**: plan にあったが実装しなかったもの → なぜ落としたか／後続 plan に回したかを記録

4. **plan ファイルに記録**
   - 「実装と計画の差分（recap）」セクションに以下のフォーマットで追記:
     ```
     ### 意図的な変更
     - <plan 上の項目>: <変更後> — <経緯>

     ### 想定外の追加
     - <追加されたもの>: <経緯>

     ### 削除・スキップ
     - <落としたもの>: <理由>（後続: <ある場合>）
     ```
   - 経緯は短くてよい（1 行〜2 行）。「学び」が出てきた場合はそれも 1 行で残す

5. **status と updated を更新**
   - `status: done`
   - `updated`: 当日

6. **user に提示**
   - 差分が大きい項目だけを 1〜3 行で要約
   - 後続 plan に切り出すべき候補があれば slug 案を添えて提示
   - 「`<slug>` の workflow を完了しました」で締める

## 注意

- recap は「責任追及」ではなく「未来の自分への記録」。書きすぎず、判断のキー要素だけを残す
- 大きな乖離が見つかった場合、SKILL.md / 各フェーズ MD にフィードバックがあるか考え、改善提案を user に出してもよい
