## 目的・背景

マイルストーン 1 の最後の未実装コマンド。`nob show <title>` で vault 上のノートのメタ情報（パス、サイズ、文字数、frontmatter）を表示する。

design.md L207-221 の出力例に倣うが、`---links---` セクション（リンク解析・未解決判定）は本 plan のスコープ外とする（user 判断、phase 2 以降）。

`create` / `list` と同じく `lib/nob/notes/` 配下にオペレーションを追加し、`Notes::Creator`/`Notes::Lister` と一貫した API・命名にする。

## 設計

### モジュール構成

```
lib/nob/notes/viewer.rb     # Nob::Notes::Viewer — show オペレーション
spec/nob/notes/viewer_spec.rb
```

`lib/nob/cli.rb` に `show` サブコマンドを追加。

### 公開 API

```ruby
Nob::Notes::Viewer.show(vault:, title:)
  # => Nob::Notes::Viewer::Result
```

```ruby
Result = Struct.new(
  :absolute_path,  # String, 絶対パス
  :relative_path,  # String, vault 相対
  :size,           # Integer, バイト数
  :chars,          # Integer, 本文（frontmatter 除く）の文字数
  :frontmatter,    # Hash, YAML としてパース済み（無ければ {}）
  keyword_init: true
)
```

### title 解決ルール

- vault 全体から basename 一致（`<title>.md`）でファイルを検索
- ドットファイル / ドットディレクトリは除外（`Notes::Lister` と同じ規則）
- 0 件 → `Nob::Notes::Viewer::NotFound < Nob::Error`
- 1 件 → そのファイル
- 複数件 → `Nob::Notes::Viewer::Ambiguous < Nob::Error`（メッセージに候補の relative_path を列挙）

### CLI 出力フォーマット

design.md の例から links セクションを除いた形：

```
Path     : daily/2024-03-27.md
Size     : 1.2KB
Chars    : 1024
---frontmatter---
title    : 2024-03-27
date     : 2024-03-27
```

- `Path` は vault 相対パス（`relative_path`）
- `Size` は人間可読（後述の Size 整形ルール参照）
- `Chars` は本文（frontmatter 除く）の文字数
- `---frontmatter---` セクションは frontmatter が非空のときのみ出力。値は YAML 上の登場順を保つ
- 他の Nob::Error と同様、エラー時は `warn "Error: #{e.message}"; exit 1`

#### Size 整形ルール

- 1024 未満 → `<n>B`（例: `512B`）
- 1024 以上 1024² 未満 → `<n.n>KB`（例: `1.0KB`, `1.2KB`）
- 1024² 以上 → `<n.n>MB`（例: `3.4MB`）
- KB/MB は小数点 1 桁、`Float#round(1)` で丸め（半数切り上げの Ruby 既定挙動）
- 境界例: `1024 → 1.0KB`, `1023 → 1023B`, `1536 → 1.5KB`

#### Ambiguous メッセージ例

```
multiple notes match "Plan": projects/Plan.md, archive/Plan.md
```

- 候補は `relative_path` をカンマ + 半角スペースで列挙、ソート順は basename → relative_path 昇順

### 設計判断メモ

- **命名**: `Notes::Viewer` を採用。既存 `Notes::Creator` / `Notes::Lister` と動詞 -er で揃える
- **戻り値**: `Nob::Entities::Note` ではなく `Notes::Viewer::Result` 専用 Struct。`Entities::Note` は「vault 上に存在するノートの値オブジェクト」だが、Viewer の戻り値は「ノートを開いて取り出した整形済み情報」で性質が違う（`Creator::Result` と同じ位置づけ）
- **Result のフィールド名**: `path` ではなく `absolute_path` を採用し `Entities::Note` と揃える（同じ「ノートのパス」概念で命名がブレるのを防ぐ）。`Creator::Result` の `path` は本 plan のスコープ外で、揃えるなら別タスクとして後続で扱う
- **Chars の定義**: 本文（frontmatter を完全に剥がした残り）の `String#length`。具体的には `front_matter_parser` の `content` 戻り値の長さを使う（後者の挙動: frontmatter ブロック `---...---` 全体と、その直後の改行 1 つまでを剥がした残りの文字列を返す）。frontmatter は別セクションで全表示するので、本文と二重カウントしないほうが読み手に有用
- **Size の定義**: `File.size`（バイト数）。人間可読の整形は CLI 層の責務（整形ルールは「CLI 出力フォーマット」参照）
- **frontmatter のパース**: 既存 gemspec 依存の `front_matter_parser` を利用（`Creator` の YAML.dump と対応）
- **frontmatter なし**: `front_matter_parser` は frontmatter のないファイルに対して空 Hash を返す。`frontmatter == {}` として扱い、CLI は `---frontmatter---` セクションごと省略する
- **frontmatter が壊れている（YAML パース不能）**: `Nob::Notes::Viewer::InvalidFrontmatter < Nob::Error` を raise。CLI は他の Nob::Error と同様に `warn "Error: ..."; exit 1`。理由 — frontmatter があるのに正しく読めない状態は user に通知すべきデータ問題で、サイレントに空 Hash 化するとデバッグしにくい
- **frontmatter Hash の順序**: Ruby Hash は挿入順を保つ。`front_matter_parser` (~> 1.0) も挿入順を保持する実装になっている（実装時に挙動を spec で確認する）
- **検索範囲**: vault 全体（`Lister` のような `prefix` 絞り込みは要求にないため見送り）
- **basename 一致の glob 方針**: `Dir.glob("**/*.md", base: vault)` で全 .md を取り、`File.basename(path, ".md") == title` で絞り込む。`**/<title>.md` のように title を glob パターンに埋めると、title にメタ文字（`*` `?` `[`）が含まれる場合に誤動作する
- **複数ヒット時**: `Ambiguous` で候補列挙。CLI も同じメッセージをそのまま `warn` する（曖昧なら user に明示するよう促す方針、design.md L202）
- **スコープ外（明示）**: `[[]]` リンク抽出、リンク解決、未解決判定。本 plan には含めない。後続で必要になったら別 plan（仮: `note-links`）で扱う

### spec の方針

`spec/nob/notes/viewer_spec.rb`:

- 単一ヒット（frontmatter なし）— absolute_path / relative_path / size / chars / frontmatter == {} を確認
- 単一ヒット（frontmatter あり）— frontmatter Hash の中身、Chars が本文（`---` デリミタを含まず、本文部分のみ）の長さであること
- frontmatter Hash のキー順が YAML 上の登場順を保つこと
- 壊れた frontmatter → `InvalidFrontmatter`
- 0 件 → NotFound（メッセージに title）
- 複数件 → Ambiguous（メッセージに候補の relative_path 全件、basename → relative_path 昇順）
- ドット隠蔽（`.hidden.md`, `.cache/x.md` はヒット対象外）
- ネストしたディレクトリでも basename で見つかる
- title にメタ文字（`*`, `?`）を含むケースでも誤マッチしない

`spec/nob/cli_spec.rb`:

- `nob show <title>` の stdout 出力フォーマット（Path/Size/Chars/frontmatter セクション）
- Size の境界整形（`512` バイト → `512B`、`1024` バイト → `1.0KB`、`1024*1024` → `1.0MB`）
- frontmatter が空のときに `---frontmatter---` セクションを出さない
- NotFound / Ambiguous / InvalidFrontmatter 時に `warn` + exit 1（既存の Nob::Error rescue パターンの確認）

### 計画から外れた / 計画になかった対応

- **`front_matter_parser` の Date 許可**: 既定の YAML loader は `Psych::DisallowedClass: Date` を投げるため、`FrontMatterParser::Loader::Yaml.new(allowlist_classes: [Date, Time])` を `YAML_LOADER` 定数として導入。Creator が `Date` を frontmatter に書く以上、Viewer 側で読み戻せないと一貫性が崩れる。
- **`Chars` の境界定義微修正**: plan で「`---` デリミタ全体と直後の改行 1 つを剥がした残り」と書いたが、実装では `front_matter_parser` の `parsed.content` をそのまま採用。挙動は spec で固定（`"body here\n".length` 等）。
- **CLI spec を tmpdir ベースに分離**: 「frontmatter なしのとき `---frontmatter---` を出さない」ケースは fixture vault では既存ノート全てが frontmatter 付きのため検証不能。`@vault = Dir.mktmpdir` ベースの context を `before/after` で組み、`allow(Nob::Config).to receive(:default_path)` で差し替え。
- **rspec lifecycle**: 当初 `around` で stub したが rspec-mocks の double がブロック終了時にクリーンアップされ後段 spec で「double outside example」エラー。`before/after` に切り替え。
- **空ファイル境界 spec 追加（Refine）**: code review の Nice-to-have を取り込み、`size == 0` / `chars == 0` を確認する spec を viewer_spec.rb に追加。
- **ドット隠蔽 spec の刷新（Refine）**: 当初 `.cache/visible.md` を `visible` で検索しトートロジー指摘。`.cache/note.md` を `note` で検索する形に書き直し、ドット隠蔽が無ければヒットするはずのケースで NotFound になることを検証。

### 学び

- **既存依存の挙動は事前確認が安い**: Date 許可、`content` の trailing newline、Hash 順序はいずれも実装中に発覚。今後 `front_matter_parser` を使う追加コマンド（show 系の拡張、リンク解析など）でも同じ前提に乗れる。
- **fixture vault は「全ノートが frontmatter 付き」など暗黙の不変量を持つ**: 個別ケース検証は tmpdir で組むのが安全。fixture を肥大化させない。
- **rspec-mocks lifecycle と `around` の相性**: double を貼る context は `before/after` を使う。`around` を使うのは「実行順を本当に制御したい」ときだけ。
